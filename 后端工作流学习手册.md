# SysMLDocGen 后端工作流学习手册

> 按**真实业务顺序**介绍后端如何运转、模块如何联动、关键函数做什么。  
> 每一章末尾有**自测题**，用于答辩前确认你是否真正理解。  
> 建议配合 `后端零基础导读.md` 对照源码阅读。

---

## 总览：一条完整业务链

```
用户登录（认证）
    ↓
创建/进入项目（项目管理 + 权限）
    ↓
上传 SysML 模型文件（模型 API）
    ↓
格式校验 → 解析 → 写入元素表（模型服务链）
    ↓
准备/选择文档模板（模板管理）
    ↓
选项目+模型+模板 → Jinja2 渲染 → 存文档（文档生成）
    ↓
下载 Word/PDF（文档导出）
```

**横向支撑（每一步都可能用到）**：

- `get_db`：数据库会话  
- `get_current_user`：JWT 识别当前用户  
- `project_access`：项目/模型读写权限  
- `create_log`：记录操作日志（管理员可查）

## 第 0 步：程序如何启动（一切的基础）

### 0.1 这一步在干什么

运行 `uvicorn` 后，`main.py` 加载所有路由，连接 MySQL，创建存储目录和数据库表。

### 0.2 关键文件与函数

| 文件 | 函数/代码 | 作用 |
|------|-----------|------|
| `app/main.py` | `app = FastAPI(...)` | 创建 Web 应用 |
| `app/main.py` | `include_router(...)` × 6 | 挂载 auth、projects、models、templates、documents、admin |
| `app/main.py` | `on_startup()` | 建 `storage/*` 目录，调用 `init_db()` |
| `app/database.py` | `init_db()` | `create_all()` 根据 ORM 建表 |
| `app/database.py` | `get_db()` | 每个请求分配一个 `db`，结束关闭 |
| `app/core/config.py` | `settings` | 数据库密码、JWT 密钥、文件路径 |

### 0.3 与其他部分的关系

- **所有 API** 都依赖 `get_db` 访问 MySQL。  
- **所有需登录的 API** 都依赖 `deps.get_current_user`。  
- 启动时 `import app.models` 确保 `project_members` 等表能被创建。

### 0.4 自测题

1. 为什么 `get_db` 要用 `yield` 而不是 `return db`？  
2. `init_db()` 和 `database/init.sql` 各在什么场景用？  
3. 六大路由模块分别对应哪六个业务域？

<details>
<summary>参考答案（先自己做，再点开）</summary>

1. `yield` 让 FastAPI 在请求结束后执行 `finally` 里的 `db.close()`，避免连接泄漏。  
2. `init.sql` 可手动建库+初始 admin；`init_db()` 在应用启动时按 ORM 自动补全/创建表（如 `project_members`）。  
3. auth、projects、models、templates、documents、admin。
</details>

## 第 1 步：用户认证（谁能访问系统）

### 1.1 这一步在干什么

用户注册/登录 → 密码 bcrypt 校验 → 签发 JWT → 之后每次请求 Header 带 `Bearer token` → `get_current_user` 解析出 `User` 对象。

### 1.2 工作流顺序

```
POST /auth/login
  → _login_with_credentials
      → verify_password
      → create_access_token({"sub": user_id})
      → create_refresh_token
      → create_log(登录成功)

后续任意 API
  → Depends(get_current_user)
      → decode_access_token
      → db.query(User).filter(id==sub)
```

### 1.3 关键文件与函数

| 文件 | 函数 | 作用 |
|------|------|------|
| `core/security.py` | `get_password_hash` | 注册/建用户时加密密码 |
| `core/security.py` | `verify_password` | 登录比对 |
| `core/security.py` | `create_access_token` | 签发短期通行证（默认约 2 小时） |
| `core/security.py` | `decode_access_token` | 校验 Token，失败返回 None |
| `core/deps.py` | `get_current_user` | 无 Token / 无效 → 401 |
| `core/deps.py` | `require_role(*roles)` | 限制 admin/manager 才能调用的接口 |
| `api/auth.py` | `register` | 新建用户，默认 role=member |
| `api/auth.py` | `login` / `login_token` | 返回 Token |
| `api/auth.py` | `refresh_tokens` | 用 refresh_token 换新 access |

### 1.4 与其他部分的联动

| 联动对象 | 关系 |
|----------|------|
| 项目管理 | `create_project` 需要 `require_role(admin, manager)` |
| 模型/文档 | 只需 `get_current_user`，再结合 `project_access` |
| 日志 | 登录成功/失败写 `logs` 表 |
| 管理后台 | `admin/logs` 需要 `require_role(admin)` |

### 1.5 自测题

1. 密码为什么存 `password_hash` 而不是明文？  
2. JWT 的 `sub` 字段里存的是什么？  
3. member 用户登录后，能否调用 `POST /projects` 创建项目？为什么？  
4. Token 过期后，前端 refresh 接口调用的是哪个后端函数？

<details>
<summary>参考答案</summary>

1. bcrypt 单向哈希，数据库泄露也无法还原密码。  
2. 用户 id（字符串形式）。  
3. 不能；`create_project` 要求 admin 或 manager。  
4. `auth.refresh_tokens`，内部 `decode_refresh_token` 后重新 `create_access_token`。
</details>

## 第 2 步：项目管理（业务数据的容器）

### 2.1 这一步在干什么

项目是顶层容器：模型挂在 `project_id` 下，文档生成也要指定 `project_id`。同时通过 `owner_id` 和 `project_members` 控制谁能看、谁能改。

### 2.2 工作流顺序

```
POST /projects（admin/manager）
  → Project(name, owner_id=当前用户)
  → db.add → commit
  → create_log

GET /projects
  → member：JOIN project_members 过滤可见项目
  → admin/manager：查全部

上传模型 / 生成文档 之前
  → require_project_write(project_id)  或  require_project_read
```

### 2.3 关键文件与函数

| 文件 | 函数 | 作用 |
|------|------|------|
| `api/projects.py` | `list_projects` | 列表；member 只看自己的+被加入的 |
| `api/projects.py` | `create_project` | 仅 admin/manager |
| `api/projects.py` | `get_project` | `can_view_project` |
| `api/projects.py` | `add_member` | 把 member 用户加入项目（只读协作） |
| `api/projects.py` | `remove_member` | 移除成员 |
| `core/project_access.py` | `can_write_project` | admin/manager 全写；member 仅 owner 可写 |
| `core/project_access.py` | `can_view_project` | owner 或成员或 admin/manager 可读 |
| `core/project_access.py` | `require_project_write` | 写操作前调用，否则 403 |
| `core/project_access.py` | `require_project_read` | 读操作前调用 |
| `core/project_access.py` | `require_model_access` | 通过模型的 project_id 间接校验 |

### 2.4 与其他部分的联动

| 联动对象 | 关系 |
|----------|------|
| 模型管理 | 每个 `SysModel.project_id` 必须指向存在的项目；上传前 `require_project_write` |
| 文档生成 | `generate_document` 校验 `project_id` 且 `model.project_id` 一致 |
| 用户表 | `projects.owner_id` → `users.id`；`project_members.user_id` → `users.id` |
| 认证 | 先登录拿到 User，再判断 role 和 owner |

### 2.5 自测题

1. 项目 owner 和 `project_members` 里的 member，在上传模型权限上有何区别？  
2. `require_project_write` 和 `require_project_read` 分别在什么场景用？各举一个本项目的 API。  
3. 为什么 `add_member` 不允许添加 admin 用户？  
4. 删除项目后，其下模型数据会怎样？（看数据库外键设计）

<details>
<summary>参考答案</summary>

1. owner（或 admin/manager）可写；被加入的 member 只能读，不能上传/生成文档。  
2. write：上传模型、生成文档；read：查看列表、预览文档、查元素树。  
3. 设计上 member 是协作只读角色，管理员用 admin 账号即可。  
4. `models` 表对 `projects` 有 `ON DELETE CASCADE`，模型及元素会级联删除。
</details>

## 第 3 步：模型上传与解析（把 SysML 文件变成数据库里的元素）

### 3.1 这一步在干什么

用户上传 `.xmi/.xml/.json/.zip` → 文件存磁盘 → 数据库 `models` 表一条记录 → 触发解析 → 结果写入 `model_elements`、`model_relationships`。

### 3.2 工作流顺序（核心链）

```
api/models.py  upload_model 或 upload_complete
    │
    ├─ require_project_write
    ├─ validate_extension / validate_size
    ├─ 文件写入 storage/models/{uuid}.xmi
    ├─ 创建 SysModel(parse_status=pending)
    │
    └─ 若 auto_parse 且 run_async
           BackgroundTasks → parse_tasks.run_parse_from_disk
       否则
           model_import_service.import_model_from_disk
                │
                ├─ format_validator.validate_and_load_payload  （ZIP 解压+魔数）
                ├─ model_parser.parse_model_bytes               （XML/JSON→列表）
                └─ 按 import_mode 写 model_elements / model_relationships
                └─ 更新 parse_status = success / failed
```

### 3.3 分层与函数表

**API 层 `api/models.py`**

| 函数 | 作用 |
|------|------|
| `upload_model` | 直传整个文件 |
| `upload_init` / `upload_chunk` / `upload_complete` | 大文件分片上传 |
| `parse_model` | 对已存在文件重新解析 |
| `list_models` / `get_model` / `delete_model` | 查询与删除 |
| `list_elements` / `get_element_tree` | 给前端展示解析结果 |
| `update_element` | 人工改元素名称/描述 |

**格式校验 `services/format_validator.py`**

| 函数 | 作用 |
|------|------|
| `validate_extension` | 只允许 xmi/xml/json/zip |
| `validate_size` | 最大 500MB |
| `validate_and_load_payload` | ZIP 安全解压，取出内层模型字节 |
| `_zip_entry_is_safe` | 防 `../` 路径穿越 |

**解析器 `services/model_parser.py`**

| 函数 | 作用 |
|------|------|
| `parse_model_bytes` | 按扩展名分流 |
| `_parse_xmi_xml` | lxml 遍历，收集 xmi:id、关系 |
| `_parse_json` | 读 elements/relationships 数组 |

**导入服务 `services/model_import_service.py`**

| 函数 | 作用 |
|------|------|
| `import_model_file` | 校验→解析→写库，返回 (bool, msg) |
| `import_model_from_disk` | 从 `model.file_path` 读文件再导入 |

**异步 `services/parse_tasks.py`**

| 函数 | 作用 |
|------|------|
| `run_parse_from_disk` | **新建** `SessionLocal()`，避免与请求会话冲突 |
| `run_parse_from_bytes` | 刚上传后用内存字节解析 |

### 3.4 与其他部分的联动

| 联动对象 | 关系 |
|----------|------|
| 项目管理 | 上传前 `require_project_write(project_id)` |
| 文档生成 | 从 `model_elements` 读数据填 Jinja2 模板 |
| 认证 | 需登录；`require_model_access` 查所属项目权限 |
| 日志 | 上传/解析成功记 operation 日志 |

### 3.5 import_mode 说明

| 模式 | 行为 |
|------|------|
| `replace` | 删掉该模型下全部元素和关系，再插入新的 |
| `merge` | 同 element_id 更新，新的追加；关系表先删再插 |
| `new` | 与 replace 相同（首版实现） |

### 3.6 自测题

1. 为什么异步解析必须用 `parse_tasks` 里新的 `SessionLocal()`？  
2. 上传 ZIP 文件时，`format_validator` 如何选出要解析的内层文件？  
3. `parse_status` 从 pending 到 success 之间，代码在哪一步设为 `parsing`？  
4. 解析失败时，元素表会不会留下半套脏数据？（replace 模式下）  
5. `get_element_tree` 是在 API 层组树，还是在 `model_parser` 里组树？

<details>
<summary>参考答案</summary>

1. HTTP 请求结束后原 `db` 已关闭，后台任务需独立会话。  
2. 优先级：`.xmi` > `.xml` > `.json`，且路径必须安全。  
3. `import_model_file` 开头：`model.parse_status = PARSING`，然后 `commit`。  
4. replace 模式会先 delete 再 insert；若解析失败则不会执行 insert，只更新 failed 状态。  
5. 在 `api/models.py` 的 `get_element_tree`：查平铺列表后按 `parent_element_id` 挂 children。
</details>

## 第 4 步：模板管理（文档长什么样）

### 4.1 这一步在干什么

模板 = 存在数据库 `templates.content` 里的一段 **HTML + Jinja2 占位符**。不解析模型，只负责「版式」。

### 4.2 工作流顺序

```
POST /templates
  → Template(name, template_type, content, creator_id)
  → db.add → commit

生成文档时（下一步）
  → documents.py 读取 Template.content
  → Jinja2 渲染
```

### 4.3 关键文件与函数

| 文件 | 函数 | 作用 |
|------|------|------|
| `api/templates.py` | `list_templates` | 全局模板列表（登录即可） |
| `api/templates.py` | `create_template` | 新建 |
| `api/templates.py` | `update_template` | 改 content/状态等 |
| `api/templates.py` | `delete_template` | 删除 |
| `scripts/seed_templates.py` | `seed()` | 插入演示用「需求规格说明书」模板 |

### 4.4 与其他部分的联动

| 联动对象 | 关系 |
|----------|------|
| 文档生成 | `generate_document` 用 `template_id` 取 `content` |
| 认证 | 需登录；创建者记入 `creator_id` |
| 模型解析 | **无直接调用**；模板里用 `{{ model_name }}`、`{% for e in elements %}` 等占位，数据来自文档模块组装的 context |

### 4.5 自测题

1. 模板内容存在磁盘 `storage/templates` 还是数据库？实际生成用哪个？  
2. 修改模板后，已生成的旧文档会自动更新吗？  
3. `seed_templates.py` 的作用是什么？答辩演示前常先做哪一步？  
4. 模板里的 `elements` 变量是谁、在哪个文件里塞进 context 的？

<details>
<summary>参考答案</summary>

1. 主要在数据库 `templates.content`；`file_path` 字段当前基本不用。  
2. 不会；需重新 `POST /documents/generate`。  
3. 插入演示模板；`python -m scripts.seed_templates`。  
4. `documents.generate_document` 从 `ModelElement` 查库后组装 `context["elements"]`。
</details>

## 第 5 步：文档生成与导出（产出最终文档）

### 5.1 这一步在干什么

用户指定 **project_id + model_id + template_id** → 校验权限与归属 → 查全部模型元素 → 拼 Jinja2 上下文 → 渲染 HTML → 存入 `documents` 表 → 需要时再转 docx/pdf。

### 5.2 工作流顺序

```
POST /documents/generate
    │
    ├─ require_project_write(project_id)
    ├─ 查 SysModel，校验 model.project_id == project_id
    ├─ require_model_access(write=False)  // 读模型即可
    ├─ 查 Template
    ├─ 查全部 ModelElement
    ├─ 构建 context（含 elements、get_element_by_id 等函数）
    ├─ SandboxedEnvironment.from_string(template.content).render(**context)
    ├─ Document(content=HTML, status=success) → db.add
    └─ create_log

GET /documents/{id}/download?fmt=docx|pdf
    ├─ require_project_write
    ├─ html_to_docx 或 xhtml2pdf/weasyprint
    └─ FileResponse 返回文件
```

### 5.3 关键文件与函数（均在 `api/documents.py`）

| 函数 | 作用 |
|------|------|
| `generate_document` | 核心：渲染并入库 |
| `list_documents` | 分页列表 |
| `preview_document` | 返回 HTML 字符串 |
| `download_document` | 转 docx/pdf 并下载 |
| `html_to_docx` | BeautifulSoup + python-docx |
| `_prepare_html_for_pdf` | 注入中文字体 CSS |
| `get_element_by_id` 等（闭包） | 模板内按 id/模块名查询元素 |

**context 里有什么（模板可直接用）**

| 键 | 来源 |
|----|------|
| `project_id`, `model_name`, `model_version`, `template_name`, `generate_time` | 元信息 |
| `elements` | 全部元素的 name/type/description 列表 |
| `get_element_by_id(id)` | 闭包，再查库 |
| `get_requirements_by_module(name)` | Requirement 且名称包含 name |
| `get_blocks_by_package(name)` | Block 且名称包含 name |

### 5.4 与其他部分的联动

| 联动对象 | 关系 |
|----------|------|
| 项目管理 | `require_project_write`；模型必须属于该项目 |
| 模型解析 | **强依赖** `model_elements` 有数据且 parse_status=success |
| 模板管理 | 读 `Template.content` |
| 认证 | 需登录；下载也要 write 权限 |

### 5.5 自测题

1. 生成阶段存的是 HTML 还是 docx？docx 什么时候产生？  
2. 为什么用 `SandboxedEnvironment` 而不是普通 Jinja2 `Environment`？  
3. 若 Jinja2 渲染报错，数据库里会不会有 document 记录？`status` 是什么？  
4. `generate_document` 里为什么要检查 `model.project_id != doc_in.project_id`？  
5. 渲染失败后，用户应看哪个字段排查模板语法问题？

<details>
<summary>参考答案</summary>

1. 生成只存 HTML 到 `documents.content`；下载时 `download_document` 才转 docx/pdf。  
2. 限制模板执行危险 Python，防止模板注入；`autoescape` 防 XSS。  
3. 会有记录，`status=failed`，`generate_message` 存异常信息。  
4. 防止用 A 项目的模型配 B 项目的 id 越权生成。  
5. `documents.generate_message` 或接口返回的失败记录。
</details>

## 第 7 步：端到端串联（答辩口述版）

按时间顺序说一遍（建议背熟）：

1. **启动**：`main.py` 注册路由，`init_db` 建表。  
2. **登录**：`auth` 校验密码，发 JWT。  
3. **建项目**：`projects.create_project`，owner 绑定当前用户。  
4. **上传模型**：`models.upload_*` → 存盘 → `import_model_file` 链 → `model_elements` 有数据。  
5. **准备模板**：`templates` 或 `seed_templates`。  
6. **生成文档**：`documents.generate_document`，Jinja2 填模板，HTML 入库。
