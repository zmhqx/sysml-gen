## 二、分模块说明（原理 · 难点 · 风险与对策 · 需求对应）

### 2.1 格式校验 `format_validator.py`

**代码原理**

- 在上传字节流或合并后的缓冲区上执行**分层校验**：扩展名白名单（`.xmi` / `.xml` / `.json` / `.zip`）→ 单文件大小上限（与概要一致的 **500MB** 量级）→ **魔数**（ZIP 以 `PK` 开头、JSON 以 `{`/`[`、XML 以 `<?xml`/`<xmi` 等）→ ZIP 内**安全解压路径**（禁止 `..`、绝对路径等路径穿越）。
- ZIP 包内按优先级选取主模型文件：**`.xmi` → `.xml` → `.json`**，再对内层内容做二次大小与类型校验。

**实现难点**

- **扩展名与真实格式不一致**（例如 OLE 的 Word 误命名为 `.docx`）：通过魔数与内层校验降低误判；ZIP 内需再验内层扩展名与内容头。
- **ZIP 炸弹 / 路径穿越**：必须在解压前检查 `namelist` 中每条路径，且仅读取选定条目而非全盘解压到不可控路径。

**主要可能问题与对策**

| 问题 | 对策/现状 |
|------|-----------|
| 恶意超大 ZIP 内单文件 | 内层 `validate_size` 再次限制 |
| 非 UTF-8 JSON | 当前按 UTF-8 解码失败会报校验错误；后续可指定编码或探测 |
| MIME 与扩展名双因子 | 概要要求 MIME；当前以魔数+扩展名为主，可再接入 `python-magic` 等 |

**需求对应**

- 对应概要设计 **4.4.1 / 4.4.3** 中 `FormatValidator`、允许类型与大小、ZIP 安全解压等描述。

### 2.2 模型解析 `model_parser.py`

**代码原理**

- **XML/XMI**：使用 `lxml` 大文档友好解析（`huge_tree`、`recover`、禁实体解析防 XXE），遍历树；对有 **`xmi:id` / `http://www.omg.org/XMI` 命名空间 id** 的节点抽取为「模型元素」；父子关系由 **DFS 遍历时栈上的 parent xmi:id** 推导。
- **关系**：对部分标签（如 Generalization、Dependency 等）读取 `client`/`supplier`/`source`/`target` 或 `general`/`specific` 等属性生成简化「关系」行（启发式，非完整 SysML 语义）。
- **JSON**：约定根对象含 `elements`（及可选 `relationships`）数组，字段映射到库表 `model_elements` / `model_relationships` 语义。

**实现难点**

- **SysML/UML XMI 方言众多**：不同工具导出的命名空间、`xmi:id` 挂点、关系表示方式不一致；通用 DOM 遍历无法一次覆盖全部规范。
- **关系成对性**：许多连接器用子元素而非属性表达端点，当前实现为**最小子集**，复杂图需按样例扩展 XPath/规则。

**主要可能问题与对策**

| 问题 | 对策/后续 |
|------|-----------|
| 解析结果为空 | 返回明确 `parse_message`；验收时用带 `xmi:id` 的样例文件 |
| 元素重复 id | 使用 `seen_ids` 去重；`merge` 模式按 `element_id` 更新 |
| 性能 | 超大模型同步解析仍耗时；已引入异步解析（见 2.6） |

**需求对应**

- 概要 **4.4.3 ModelParser**、解析产物写入持久化、与 `import_mode` 相关的存储策略（新建/覆盖/合并的简化落地）。

### 2.3 导入服务 `model_import_service.py`

**代码原理**

- **管道顺序**：校验载荷 → `parse_model_bytes` → 按 `import_mode` 写库。
- **`replace` / `new`**：删除该模型下全部 `model_relationships` 与 `model_elements` 后批量插入（首版 `new` 与 `replace` 行为一致，避免空合并语义歧义）。
- **`merge`**：按 `element_id` **存在则更新、不存在则插入**；关系仍采用「先删后插」简化以保证与本次解析文件一致。

**实现难点**

- **事务与一致性**：批量删除+插入需在单事务中完成；失败时更新 `parse_status=failed` 与 `parse_message`。
- **与异步任务协作**：异步任务使用**新 Session**（见 `parse_tasks.py`），避免与 HTTP 请求共用一个已关闭的 `Session`。

**主要可能问题与对策**

| 问题 | 对策 |
|------|------|
| 异步任务读到旧文件 | 先 `commit` 模型与文件落盘再 `add_task` |
| merge 与「按 name+type+parent」概要差异 | 当前以 `element_id` 为主键合并；后续可对齐概要算法 |

**需求对应**

- 概要中 **MdkBroker → ModelService 入库** 的简化单体实现；`import_mode` 三种策略的可用子集。

---

### 2.4 项目与模型访问控制 `project_access.py` 与路由配合

**代码原理**

- **`can_write_project`**：管理员、经理**可写任意项目**；成员仅当 `project.owner_id == user.id`。
- **`can_view_project` / `require_project_read`**：与读权限一致——成员**只能读自己拥有的项目**；经理/管理员可读全部（用于列表、文档预览、模型读等）。
- **模型写操作**（上传、解析、删模型、改元素等）：`require_model_access(..., write=True)` → 底层仍要求**项目写权限**。

**实现难点**

- **「成员不能建项目」与「成员要有可读模型」** 的平衡：若成员永远不是 `owner`，则按 owner 过滤会看不到经理创建的项目；当前验收策略是 **读自己项目、经理/管理员看全部**；与「全员可读」相比更贴近概要中的**按项目授权**。
- **经理改删他人项目**：`PUT/DELETE /projects` 仍主要校验**角色**，未再加「仅本人项目」限制，避免与原有管理流程冲突（可在后续迭代明确）。

**主要可能问题与对策**

| 问题 | 说明 |
|------|------|
| 跨项目越权 | 文档生成已校验 `model.project_id == doc_in.project_id` |
| 成员无项目 | 列表为空属预期；需经理为其创建项目并转移 owner 或扩展成员表 |

**需求对应**

- 概要 **AuthService 按项目鉴权** 的简化落地；与 **4.4.4** 文档侧 `require_project_write` / `require_project_read` 配合。

### 2.5 模型 API `api/models.py`（上传、分片、解析、列表）

**代码原理**

- **单文件上传**：读流 → 校验 → 磁盘 UUID 文件名 → 写 `SysModel` → `auto_parse` 时同步或异步调用导入服务。
- **分片上传**：`init` 写 `meta.json`（`user_id` 绑定）；`chunk` 写 `part_xxxxx`；`complete` 校验总大小、合并、落盘、建库、可选异步解析；临时目录 `storage/models/_uploads/{upload_id}` 完成后 `rmtree`。
- **`POST /{model_id}/parse`**：支持 Body 中 `import_mode` 与 **`run_async`**；异步时先将状态置为 `pending` 与提示文案，再投递后台任务。
- **列表**：`member` 无 `project_id` 时 `join Project` 过滤 **owner 为自己** 的模型。

**实现难点**

- **FastAPI 参数顺序**：`BackgroundTasks` 必须出现在带默认值参数（`Form`/`Depends`）之前，否则违反 Python 语法；已调整为 `background_tasks` 前置。
- **分片会话与多进程**：会话状态在本地目录，**多 worker 进程**下 chunk 可能落到不同机器；生产环境需对象存储或 Redis 会话。

**主要可能问题与对策**

| 问题 | 对策 |
|------|------|
| 放弃上传遗留目录 | 可加重试定时任务清理 `_uploads` |
| 异步解析失败 | 轮询 `GET /models/{id}` 看 `parse_status` / `parse_message` |

**需求对应**

- 概要 **4.4.1** 上传与大小、**4.4.3** 导入管道、性能相关「异步」方向的初版。

---

### 2.6 异步解析任务 `services/parse_tasks.py`

**代码原理**

- 在后台线程中 **`SessionLocal()` 新建会话**，查询 `SysModel` 后调用 `import_model_from_disk` 或 `import_model_from_bytes`，在 `finally` 中关闭会话，避免与请求生命周期绑定。

**实现难点**

- **不能在 `BackgroundTasks` 里直接使用请求中的 `db`**：请求结束 `get_db` 会 `close`，导致惰性加载或提交异常。

**主要可能问题与对策**

| 问题 | 对策 |
|------|------|
| 与同步双写竞争 | 同一模型避免并发多次 parse；产品可加「解析中」互斥锁或版本号 |
| 进程重启任务丢失 | 后续换 Celery/RQ + 持久化任务表 |

---

### 2.7 文档 API `api/documents.py` 与 Schema

**代码原理**

- **列表**：`GET /documents` 返回 `DocumentListResponse`（`items` + `total` + `page` + `page_size`）；支持 `project_id`；`member` 无 `project_id` 时仅列出**自己项目**下的文档。
- **生成**：`SandboxedEnvironment` 编译模板，限制危险能力；上下文注入 `get_element_by_id`、`get_requirements_by_module`、`get_blocks_by_package` 等**闭包**访问当前 `model_id` 下元素；**强制** `model.project_id == doc_in.project_id`。
- **下载**：`docx` 用 `python-docx`；`pdf` 在 WeasyPrint 可用时用 `HTML(string=...).write_pdf`；成功后回写 `file_path`、`export_format`。

**实现难点**

- **沙箱与业务函数**：通过向 context 注入可调用对象扩展能力；需控制闭包内仅访问当前模型，避免跨模型 SQL。
- **WeasyPrint 在 Windows**：常依赖 GTK 等运行时，`ImportError` 时降级为不可用并返回 **503** 说明。

**主要可能问题与对策**

| 问题 | 对策 |
|------|------|
| HTML 非完整文档结构影响 PDF | 模板侧规范外层 HTML；后续可加默认 CSS |
| 大文档列表内存 | 分页已缓解；可再加仅列表不含 `content` 的 DTO |

**需求对应**

- 概要 **4.4.4** 沙箱模板、HTML/PDF、项目维度、历史查询方向的「列表分页」子集。

---

### 2.8 认证 `security.py` + `api/auth.py` + 前端 `api/index.ts` / `stores/auth.ts`

**代码原理**

- **双令牌**：`create_access_token` 写入 `typ=access`，`create_refresh_token` 写入 `typ=refresh`；`decode_access_token` 若发现 `typ==refresh` 则拒绝，防止 refresh 误作 Bearer 调业务接口。
- **刷新**：`POST /auth/refresh` 校验 refresh JWT 与用户状态，签发**新的 access 与 refresh**（旋转 refresh，降低泄露窗口）。
- **前端**：登录保存 `refresh_token`；响应拦截器在 **401** 时对非 login/refresh 请求尝试 refresh 并重试原请求。

**实现难点**

- **401 与业务「未授权」**：需排除登录、刷新路径防止死循环；并发 401 时用 `isRefreshing` 标志降低风暴（简化版）。

**主要可能问题与对策**

| 问题 | 对策 |
|------|------|
| 旧 token 无 `typ` 字段 | `decode_access_token` 仅拒绝明确为 refresh 的载荷，旧 access 仍可用直至过期 |
| XSS 窃取 localStorage | 生产应 HttpOnly Cookie + CSRF 或缩短链路由网关统一鉴权 |

**需求对应**

- 概要 **3.2** 中 JWT 时效与 refresh 的简化实现。

---

### 2.9 项目列表 `api/projects.py`

**代码原理**

- `GET /projects`：`member` 增加 `filter(owner_id==current_user.id)`；`GET /projects/{id}` 使用 `can_view_project` 防止成员窥探他人项目。

**实现难点**

- 与「经理代建项目、成员参与」的完整业务相比，缺少 **project_members** 表时只能用 **owner** 表达成员边界。

## 五、完整验收演示步骤

以下假设：**MySQL 已按 `database/init.sql` 初始化**；后端 `uvicorn` 已启动；前端 `npm run dev` 已启动（若只验后端可用 **curl / Apifox**）。

### 5.1 环境与账号

1. 确认 `backend` 可连 MySQL，`.env` 或默认配置中库名、账号正确。
2. 使用已有管理员或先 **`POST /api/v1/auth/register`** 注册成员账号，再用 **`POST /api/v1/auth/login`** 登录经理/管理员账号（创建项目需经理或管理员）。

### 5.2 认证与 Refresh

1. **`POST /api/v1/auth/login`**，记录返回的 `access_token`、`refresh_token`。
2. 使用 `Authorization: Bearer {access_token}` 调用 **`GET /api/v1/auth/me`**，应返回当前用户。
3. （可选）等待 access 过期或手动改短本地配置后，使用 **`POST /api/v1/auth/refresh`**`，Body 为 `{ "refresh_token": "..." }`，应返回新的双 token。
4. 前端：登录后打开浏览器开发者工具 → Application → Local Storage，确认存在 `token` 与 `refresh_token`；在 Network 中可模拟 401 观察是否自动 refresh（需过期或无效 access）。

### 5.3 项目可见性

1. **经理** `POST /api/v1/projects` 创建项目 A。
2. **`GET /api/v1/projects`**：经理应看到含 A 的列表；**成员**登录后应**看不到** A（除非 `owner_id` 为该成员）。
3. **成员** 访问 **`GET /api/v1/projects/{A的id}`** 应 **403**；经理访问应 **200**。

### 5.4 模型上传与解析（单文件）

1. 经理对项目 A：`POST /api/v1/models/upload`（`multipart`：`project_id`、`file`、可选 `auto_parse=true`、`run_async=true`、`import_mode=replace`）。
2. 立即 **`GET /api/v1/models/{id}`**：若异步，`parse_status` 可能为 `pending`/`parsing`，稍后刷新直至 `success` 或 `failed`，查看 `parse_message`。
3. **`GET /api/v1/models/{id}/elements`**：成功时应有条目（与样例 XMI/JSON 有关）。
4. **`POST /api/v1/models/{id}/parse`**，Body：`{ "import_mode": "merge", "run_async": false }` 验证同步路径与合并语义（需已有元素）。

### 5.5 ZIP 与校验

1. 将合法 `.xmi` 或 `.json` 打包为 `.zip` 再上传，应能解析出内层主文件。
2. 故意上传**错误魔数**或超大文件，应返回 **400** 及明确 `detail`。

### 5.6 分片上传

1. **`POST /api/v1/models/upload/init`**，Body JSON：`project_id`、`filename`、`total_chunks`、`total_size`、`name`、`version_tag`。
2. 多次 **`POST /api/v1/models/upload/chunk`**（`upload_id`、`chunk_index`、`total_chunks`、`file` 分片）。
3. **`POST /api/v1/models/upload/complete`**，核对合并后模型记录与解析状态。

### 5.7 文档生成与列表

1. **`POST /api/v1/templates`** 创建含 `{{ model_name }}` 与 `{% for e in elements %}` 的模板。
2. **`POST /api/v1/documents/generate`**，故意传 **`model_id` 不属于 `project_id`** 的项目，应 **400**。
3. 正确组合 `project_id`、`model_id`、`template_id` 生成后，**`GET /api/v1/documents?page=1&page_size=10&project_id=`** 检查分页结构与 `items`。
4. **`GET /api/v1/documents/{id}/download?fmt=html|docx|pdf`**：PDF 在无 WeasyPrint 环境下预期 **503** 或不可用说明。

### 5.8 权限回归

1. **成员** 对**非本人项目**尝试 **`POST /models/upload`**，应 **403**。
2. **成员** 对**本人项目**（若存在）上传应 **200**（若业务上成员无项目则跳过）。

### 5.9 验收通过建议标准

- 认证：登录 / me / refresh 流程正确，refresh 不能当 access 滥用。
- 模型：单文件与 ZIP、同步与异步解析至少各通一条；`parse_status` 终态可解释。
- 文档：归属校验生效；列表分页字段齐全；沙箱模板无任意代码执行（可尝试带 `os` 的模板应失败或受限）。
- 权限：项目列表/详情与模型/文档列表在成员与经理之间表现符合上文设计。
