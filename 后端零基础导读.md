# 后端零基础导读（按文件夹顺序）

> 写给：**几乎没有 Web 开发经验**，只有一点 C/C++ 基础的同学。  
> 读法：从上到下，按章节顺序看；每章先懂「这个文件是干什么的」，再看里面的函数。  
> 更偏答辩的速查表见根目录 `README.md`。

---

## 第 0 章：先弄懂 5 个词

| 词 | 大白话 |
|----|--------|
| **后端** | 在服务器上跑的程序，接收前端的请求，查数据库、算结果，再返回 JSON 或文件。 |
| **API / 接口** | 一个网址 + 一种操作。例如 `GET /api/v1/projects` = 「给我项目列表」。 |
| **路由** | 把「网址 + 方法(GET/POST)」对应到「哪个 Python 函数去处理」。 |
| **数据库** | 持久保存数据（用户、项目、模型等），本系统用 MySQL。 |
| **ORM** | 用 Python 类代表数据库表，用 `db.query(...)` 代替手写 SQL。 |

**一次请求的粗略过程**（后面每章都会用到）：

```
浏览器/前端  →  HTTP 请求（带 Token）  →  FastAPI 路由函数
    →  权限检查  →  读/写数据库  →  返回 JSON
```

## 第 1 章：项目根目录与 `backend/` 总览

```
backend/
├── requirements.txt      # 依赖库列表（pip 安装用）
├── scripts/              # 单独运行的脚本（不通过网页访问）
└── app/                  # 真正的后端程序
    ├── main.py           # 程序入口
    ├── database.py       # 连接数据库
    ├── core/             # 配置、密码、权限等「基础设施」
    ├── models/           # 数据库表对应的 Python 类
    ├── schemas/          # 接口收发的数据结构（校验用）
    ├── services/         # 复杂业务逻辑（解析模型等）
    └── api/              # 对外 HTTP 接口（最重要的一层）
```

**分层习惯**（类似 C++ 里分 ui / logic / data）：

- `api/`：接待员，只负责接 HTTP、调别人、返回结果。  
- `services/`：专家，干脏活累活（解析 XML 等）。  
- `models/`：数据表长什么样。  
- `schemas/`：请求/响应的「表格格式」。

## 第 3 章：`app/main.py`（程序入口）

**作用**：整个后端的「main 函数」。启动后监听 8000 端口，把各模块的路由挂上来。

### 里面做了什么

| 代码 | 含义 |
|------|------|
| `app = FastAPI(...)` | 创建应用对象 |
| `CORSMiddleware` | 允许前端（5173 端口）跨域访问 |
| `include_router(auth.router)` 等 6 次 | 注册 6 组 API |
| `on_startup` → `init_db()` | 启动时自动建表、建 storage 文件夹 |
| `GET /health` | 健康检查，返回 `{"status":"ok"}` |

### 函数

| 名称 | 类型 | 作用 |
|------|------|------|
| `on_startup()` | 启动钩子 | 创建 `storage/models` 等目录，调用 `init_db()` |
| `health()` | 路由 | 测试服务是否活着 |

### 原理

`import app.models` 这行是为了让所有 ORM 类注册到 SQLAlchemy，这样 `create_all()` 才能建出全部表。

### 怎么运行

```bash
cd backend
python -m uvicorn app.main:app --host 0.0.0.0 --port 8000
```

## 第 4 章：`app/database.py`（数据库连接）

**作用**：连接 MySQL，提供「一次请求用一个数据库会话」。

### 全局变量

| 名称 | 作用 |
|------|------|
| `engine` | 数据库引擎（连接池） |
| `SessionLocal` | 会话工厂，用来 `SessionLocal()` 开新连接 |
| `Base` | 所有 ORM 表的基类 |

### 函数

| 函数 | 作用 | 谁调用 |
|------|------|--------|
| `get_db()` | 生成器：创建 `db`，`yield` 给路由用，请求结束自动 `close()` | 几乎每个 API 的 `Depends(get_db)` |
| `init_db()` | `Base.metadata.create_all(engine)` 建所有表 | `main.py` 启动时 |

### 原理

一个 HTTP 请求里，多个函数共用同一个 `db`；请求结束必须关闭，否则连接泄漏。  
后台异步任务**不能**用请求里的 `db`，要在 `parse_tasks.py` 里自己 `SessionLocal()`。

## 第 5 章：`app/core/`（核心配置与安全）

### 5.1 `config.py`

**作用**：读配置（数据库密码、JWT 密钥、文件保存路径等）。

**类 `Settings`**：字段例如 `db_host`, `db_password`, `secret_key`, `storage_model_dir`…

**属性 `database_url`**：拼出 MySQL 连接字符串，给 SQLAlchemy 用。

**全局 `settings`**：其它文件 `from app.core.config import settings` 使用。

可从 `.env` 文件覆盖默认值。

---

### 5.2 `security.py`（密码与令牌）

**作用**：密码不能明文存库；登录后发 JWT 令牌。

| 函数 | 输入 | 输出 | 原理 |
|------|------|------|------|
| `get_password_hash(password)` | 明文密码 | 哈希字符串 | bcrypt 加盐，不可逆 |
| `verify_password(plain, hashed)` | 明文 + 库中哈希 | True/False | 登录时比对 |
| `create_access_token(data)` | 如 `{"sub":"用户id"}` | JWT 字符串 | 短期通行证（默认 2 小时） |
| `create_refresh_token(sub)` | 用户 id | JWT 字符串 | 长期，用来换新的 access |
| `decode_access_token(token)` | JWT | 字典或 None | 校验签名和过期 |
| `decode_refresh_token(token)` | JWT | 字典或 None | 必须是 refresh 类型 |

**JWT 是什么**：一串编码字符串，里面藏着用户 id；前端每次请求在 Header 里带 `Authorization: Bearer xxx`。

---

### 5.3 `deps.py`（依赖注入：当前用户是谁）

**作用**：FastAPI 的 `Depends` 会在进路由函数**之前**先执行这些函数。

| 函数 | 作用 |
|------|------|
| `get_current_user` | 从 Header 取 Token → 解码 → 查 users 表 → 返回 `User` 对象；失败就 401 |
| `require_role(*roles)` | 返回一个「检查函数」：当前用户角色必须在列表里，否则 403 |

**原理**：不用在每个 API 里重复写「解析 Token」，写一行 `Depends(get_current_user)` 即可。

---

### 5.4 `project_access.py`（项目权限）

**作用**：判断「这个人能不能看/改这个项目」。

| 函数 | 含义 |
|------|------|
| `get_project(db, project_id)` | 按 id 查项目 |
| `can_write_project(user, project)` | admin/manager 全能写；member 只有自己是 owner 才能写 |
| `can_view_project(db, user, project)` | admin/manager 全能看；member 要是 owner 或在成员表里 |
| `require_project_write(...)` | 不能写就抛 403 |
| `require_project_read(...)` | 不能读就抛 403 |
| `require_model_access(..., write=True/False)` | 通过模型找到项目，再检查读写 |

**答辩必记**：被 `add_member` 加进项目的 member **只能看，不能上传模型、不能生成/下载文档**（除非他是 owner）。

## 第 8 章：`app/services/`（业务逻辑层）

### 8.1 `log_service.py`

| 函数 | 作用 |
|------|------|
| `create_log(db, log_type, operator_id, module_name, operation_content, ...)` | 往 logs 表插一条记录 |
| `get_logs(db, 筛选条件, page, page_size)` | 分页查日志，admin 用 |

---

### 8.2 `format_validator.py`（上传文件安检）

**作用**：在解析之前，确认文件类型、大小、ZIP 是否安全。

| 函数/常量 | 作用 |
|-----------|------|
| `MAX_UPLOAD_BYTES` | 最大 500MB |
| `ALLOWED_EXTENSIONS` | 只允许 .xmi .xml .json .zip |
| `FormatValidationError` | 校验失败抛这个异常 |
| `validate_extension(filename)` | 看后缀 |
| `validate_size(content)` | 看字节长度 |
| `validate_and_load_payload(filename, content)` | **总入口**：若是 zip 则解压出里面的模型文件再返回字节 |
| `_zip_entry_is_safe(name)` | 防止 zip 里写 `../../etc/passwd` 这种路径 |
| `_extract_zip_payload(zf)` | 在 zip 里选 .xmi > .xml > .json 一个文件 |

**原理**：不信任用户上传的文件名和内容，先检查再解析。

---

### 8.3 `model_parser.py`（读文件内容 → 内存里的列表）

**作用**：把 XMI/XML/JSON **文件字节**变成 Python 列表，还不写数据库。

**类 `ParseResult`**：success、elements 列表、relationships 列表、errors。

| 函数 | 作用 |
|------|------|
| `parse_model_bytes(content, inner_ext)` | 根据 .json 或 .xml/.xmi 分流 |
| `_parse_xmi_xml(content)` | 用 lxml 遍历 XML 树，找带 xmi:id 的节点当元素；再找 Generalization 等当关系 |
| `_parse_json(content)` | json.loads，读 elements 和 relationships 数组 |
| `_local(tag)` | XML 标签去掉命名空间前缀 |
| `_child_name_text(elem)` | 找子节点 `<name>` 的文字 |

**原理**：SysML 工具导出的 XMI 是 XML；解析器只做「最小可用」：能抽出 id、名字、类型、父子、部分关系即可。

---

### 8.4 `model_import_service.py`（解析结果写入数据库）

| 函数 | 作用 |
|------|------|
| `import_model_file(db, model, file_content, filename, import_mode)` | 校验 → 解析 → 按模式写 elements/relationships → 改 parse_status |
| `import_model_from_disk(db, model, import_mode)` | 从 model.file_path 读文件再调用上面 |

**import_mode**：

- `replace`：删掉该模型旧元素和关系，全部插入新的。  
- `merge`：同 element_id 更新，新的追加；关系先删再插。  

流程：先把 `parse_status` 设为 `parsing` → 失败设 `failed` 和错误信息 → 成功设 `success`。

---

### 8.5 `parse_tasks.py`（后台慢慢解析）

| 函数 | 作用 |
|------|------|
| `run_parse_from_disk(model_id, import_mode)` | 新开 db 会话，查模型，调用 `import_model_from_disk` |
| `run_parse_from_bytes(...)` | 用内存里的字节解析（刚上传完时用） |

**原理**：上传接口先返回「已接收」，解析在后台跑，前端轮询 `parse_status`。

### 9.3 `models.py`（模型上传与解析）

前缀：`/api/v1/models`（文件较长，按功能记）

**分片上传三步骤**：

1. `upload_init`：检查权限和文件名，创建临时目录，写 meta.json，返回 upload_id。  
2. `upload_chunk`：每次上传一片，保存为 part_00000 等。  
3. `upload_complete`：合并所有片 → 存到 storage/models → 创建 SysModel 行 → 可选后台解析。

**直传**：`upload_model` — 一次 POST 带整个文件。

**解析**：`parse_model` — 对已存在的文件重新解析。

**查询**：`list_models`, `get_model`, `delete_model`  
**元素**：`list_elements`, `get_element_tree`, `search_elements`, `get_element`, `update_element`  
**关系**：`list_relationships`

**共同模式**：先 `get_current_user`，再 `require_project_write` 或 `require_model_access`，再操作 `db`。

## 第 12 章：四大功能串起来（用最白的话）

### 12.1 项目管理

1. 管理员登录。  
2. `create_project` 在数据库 projects 表加一行。  
3. `add_member` 在 project_members 表加「谁可以看这个项目」。  
4. 以后所有模型、文档都带 `project_id`，权限用 `project_access.py` 判断。

### 12.2 模型解析

1. `upload_model` 把文件存到 `storage/models/随机名.xmi`。  
2. `models` 表加一行，状态 pending。  
3. 后台 `run_parse_from_disk` → `import_model_file` → `validate_and_load_payload` → `parse_model_bytes`。  
4. 结果写入 `model_elements`、`model_relationships`，状态改 success。  
5. 前端 `get_element_tree` 把父子关系拼成树显示。

### 12.3 模板管理

1. `create_template` 或运行 `seed_templates.py`。  
2. content 里写 HTML，空的地方写 `{{ 变量 }}` 和 `{% for %}`。  
3. 存在 templates 表，和项目无关（全局模板库）。

### 12.4 文档生成

1. `generate_document` 传入 project_id、model_id、template_id。  
2. 从 model_elements 读出数据，塞进 context。  
3. Jinja2 把 template.content 渲染成 HTML。  
4. 存入 documents.content。  
5. 用户点下载 → `download_document` 转成 docx/pdf 文件。

## 第 14 章：初学者常见困惑

| 困惑 | 解释 |
|------|------|
| `Depends(get_db)` 是什么？ | FastAPI 自动调用 get_db，把 db 传给函数参数。 |
| `@router.post` 是什么？ | 装饰器：把下面的函数注册成 POST 接口。 |
| `db.commit()` 是什么？ | 真正执行 INSERT/UPDATE，之前只是暂存在会话里。 |
| `db.add(obj)` 是什么？ | 告诉 SQLAlchemy「准备插入这条对象」。 |
| `response_model=UserOut` 是什么？ | 返回时自动转成 UserOut 格式，隐藏密码等。 |
| 为什么有 Model 又有 Schema？ | Model 对应数据库表；Schema 对应 API 收发的 JSON，两层分开更安全。 |
| BackgroundTasks 是什么？ | 先给前端返回响应，再在后台继续执行解析，不让用户干等。 |
